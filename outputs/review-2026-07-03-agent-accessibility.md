# NEEDS SIGN-OFF — agent-accessibility reshape (build-mcp + discovery)

Proposed from the `roast` **RESHAPE (high)** verdict + the citation-verified `storm-research` briefing +
the 48-hour MCP spike, all in `outputs/vetting/2026-07-03-agent-accessibility/`. The idea (future-proof the
apps the template builds so AI agents/assistants can use them, from Chris Raroque's "AI Era" video) was
judged real but **not as one welded five-piece bundle**. The vetting split it on a durability fault line and
the spike removed the technical risk from the MCP piece (~90%+ codegen; read-only enforced by construction).
These items encode the reshape as **new skills / new capabilities**, which is why they need sign-off (per
CLAUDE.md, a new or edited skill is NEEDS SIGN-OFF, never auto-applied).

**Scope guardrails these items must honor** (from the verdict): keep everything Tier-0 / graceful-off (works
with no keys); attended for anything irreversible; keys-in-env-never-in-chat; scaffold-+-checklist for any
go-live; **read-only by default** for agent operability; **cut Siri/App Intents** (can't run at Tier-0); and
**do not add a new API layer** (Supabase already emits PostgREST/GraphQL). The spike at `spike-mcp/` is the
proven blueprint for rv-20260703-001.

**How to apply:** check a box to approve; `improve-system` applies approved items on its next run and logs
each to `outputs/change-log.md`. Leave a box unchecked to defer/decline. These are larger than a typical
fix — approving one authorizes building that skill/capability in the template's idiom (its own SKILL.md,
provenance to `raw/`, report/checklist to `outputs/`, docs, and CLAUDE.md wiring). Do not renumber or
rewrite existing ids. Recommended order: 002 (cheapest, ships now) → 001 (the operability rung) → 003 (docs).

Ship the spike's **generic `DataStore → MCP` adapter** (~54 lines, proven in `spike-mcp/src/mcp/adapter.mjs`) as a fixed skill **asset**; per app the skill generates only the small entity **catalog** (from `app/src/data/` getters or `build-backend`'s `app/src/data/store/` interface + the charter) plus ~9 lines of wiring — auto-registering `list_<plural>`/`get_<entity>` tools, each annotated `readOnlyHint: true`, with agent-tuned descriptions drafted from the charter/field names (the one non-codegen input — flag it for a quick human review). **Read-only by construction:** no code path registers create/update/delete; a write attempt returns "Tool not found" (verified in the spike) — this is how it honors "attended for anything irreversible." **Graceful-off / Tier-0:** scaffolds a keyless **local stdio** server over the mock DataStore that runs immediately (`npm run mcp`); the **remote go-live** (swap to Streamable HTTP + OAuth 2.1 + `SupabaseStore`, targeting the 2026-07-28 final MCP spec) is a **keyed checklist step the user owns** — write it to `outputs/mcp/<date>-<slug>/GO-LIVE.md`, never enter keys. Requires a built `app/` (uses its data layer); real remote use also requires `build-backend`. Provenance `layer: mcp` to `raw/builds/` + a `## MCP` section in `wiki/build.md` + an `applied` change-log line, matching the other build-* skills. One confirm gate; re-runnable; **attended-only — never in `maintenance-loop`, and deliberately NOT in `autopilot`** (production-ish rung leading to a keyed go-live, mirroring `deploy`/`polish` — add the exclusion note to its own SKILL.md, change no autopilot file). Add it as the agent-accessibility rung in `docs/PATH-TO-PRODUCTION.md` and one skills-list line in `CLAUDE.md`. Writes are explicitly out of scope for v1 (a later opt-in, confirmation-gated, domain-function-routed adapter).  Why: the spike proved this is low-risk codegen with a real payoff (a cross-entity get-paid/tax question answered from 4 read-only calls), and read-only-by-construction cleanly resolves the roast's only architectural objection. It's the one non-commodity angle (RLS-aware, DataStore-native, read-first) vs. the already-commoditized OpenAPI→MCP generators.

- [x] `rv-20260703-002` — add a static **discovery layer** (schema.org full entity pages + `llms.txt`) to `build-app` output  ·  target: `.claude/skills/build-app/SKILL.md` (page-generation phase) + `docs/BUILD-APP.md`  ·  detail: Have `build-app` emit, for every generated app, (1) **schema.org JSON-LD as full entity-page structured data** — structured markup **plus** the contextual prose on the page, since the verified study (arXiv 2603.10700) shows JSON-LD *alone* is only marginal while a full entity-page format lifts RAG accuracy ~+29.6% — and (2) a static `public/llms.txt` describing the app's purpose, routes, and data model. Both are static, keyless, Tier-0, generated at build. **Honest framing is mandatory** (matches the template's multi-target-honesty brand): docs must say schema.org *aids AI/search understanding* — **not** "required for AI visibility" (Google's own docs say it isn't) — and must label `llms.txt` a low-cost hedge with limited confirmed consumption (~408 of ~515M AI-bot requests), never "agent-accessible." Preserve `build-app`'s clean-ship guarantee: additive only, no behavior change to existing pages, no new keys/deps beyond static output.  Why: schema.org is the one durable-now, genuinely-AI-consumed piece the council was unanimous on, and it pays off in the present tense (Google rich results + measurably higher RAG citation) independent of whether the MCP/agent bet matures; `llms.txt` is nearly free option-value. Cheapest, lowest-risk piece — ship first.
