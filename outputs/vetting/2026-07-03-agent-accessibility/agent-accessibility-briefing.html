01

The 60-Second Summary

"Make your app usable by AI agents" is a real, durable direction — but it is not one feature, it is five bets with wildly different half-lives, and lumping them together is the mistake. Two are safe to build on now: the pieces that passed to neutral, multi-vendor governance before they hit scale are the ones that will still be standing — MCP (donated to the Linux Foundation's Agentic AI Foundation in December 2025) and schema.org (multi-vendor since 2011) pass that test; Apple's App Intents/Siri does not, and it is replaying the Alexa Skills / Google Actions graveyard. The catch is that MCP is simultaneously the most durable standard and the most dangerous thing to actually turn on: the NSA published an MCP security advisory in May 2026, and rigorous work shows database-layer access control (RLS) does not stop a prompt-injected agent, because the injection happens above the database. Meanwhile the two "cheap" pieces are cheaper than advertised: schema.org genuinely helps AI understanding (Google itself uses it) but is not required for AI visibility and the viral "2.5x citations / 16%→54% accuracy" numbers are unverified marketing; llms.txt is nearly free but essentially unread (≈408 of 515M AI-bot requests). The move is to split discovery (ship now, static, keyless) from operability (a backend-gated skill built to the standard but shipped read-only until the user opts in) — which is exactly this template's existing graceful-off idiom.

03

The Hidden Connection

Two findings look like a contradiction: MCP is the safest long-term bet (Finding 1 — neutral governance, multi-vendor gravity) and simultaneously the most dangerous thing to ship (Finding 2 — NSA advisory, RLS doesn't contain prompt injection). How can the same technology be both?

Because durability and safety are orthogonal axes measuring different objects. Governance predicts whether the standard will still exist in five years. Security predicts whether a running instance can be hijacked today. The template can sit on the correct side of both at once — by generating to the durable standard while shipping the instance switched off: a read-only, typed MCP server built to the July-2026 spec, scaffolded but not live, that only turns on past a keyed go-live and an auth gateway the user owns.

Adopt the governed standard now to future-proof; ship it read-only and off to stay safe. That is not a compromise — it is precisely this template's existing graceful-off, scaffold-plus-checklist idiom, applied to a new surface.

The assumption this briefing rests on (and the missing 6th lens)

All five lenses evaluated the builder's interest — durability, security, cost, market, history. Every one froze the same variable: the end user whose data the agent would read and write. The missing sixth lens is the Privacy/Consent perspective: do the app's actual users want an AI agent able to act on their data, and what consent and liability model does that require?

This omission could invert the conclusion for consumer apps. If users don't consent — or regulation demands explicit, per-action opt-in — then even a perfectly governed, perfectly secured MCP server is unwanted attack surface, not a feature. Build the operability layer with per-user opt-in and visible consent as first-class, not an afterthought.

04

What To Actually Do Differently

For the template maintainer deciding what to build — six specific moves, in priority order.

01
Ship schema.org as full entity-page output in build-app now — not bare JSON-LD.

The verified +29.6% RAG lift comes from structured markup plus contextual prose, not tags sprinkled on. Static, keyless, durable. Frame it honestly in docs: "aids AI understanding," never "required for AI visibility" (Google says it isn't).

02
Generate a static llms.txt at build as a labeled, near-zero-cost hedge.

Two lines, no downside, first-mover on the B2A convention. But label it as speculative and never market it as "agent-accessible" — consumption is ~408 in 515M requests today. Honors the template's multi-target-honesty brand.

03
Make the MCP skill generate a READ-ONLY, typed server against the July-2026 spec — scaffolded off.

It compiles and documents intent but wires up only past a keyed go-live + auth gateway. Default to list/get tools. Gate create/update/delete behind explicit opt-in + per-operation confirmation, routed through domain/service functions (never raw DataStore.create) so domain validation and "attended for irreversible" both survive. This resolves the Finding-2 collision.

04
Cut App Intents / Siri from scope; leave an honest "native build required" note.

It can't run at Tier-0 (Expo PR #47207 unmerged, needs EAS + Apple Developer account), it's single-vendor, and Siri itself is slipping. Make it a build-mobile roadmap note; revisit when the Expo module merges and personalized Siri actually ships.

05
Split the work along the discovery↔operability fault line.

Discovery (schema.org + llms.txt): static, Tier-0, ship now — fold into build-app/polish. Operability (agentify/build-mcp): backend-prerequisite, its own clearly-labeled rung on the path-to-production ladder. This matches the template's one-skill-per-surface principle.

06
Lead with the one thing that isn't already commoditized: RLS-aware, DataStore-native, read-first wiring.

OpenAPI→MCP generation is already a solved commodity (FastMCP, Speakeasy, Supabase's own MCP server). "We generate an MCP server" differentiates nothing. "We generate a safe-by-construction, owner-scoped, read-first agent surface from a uniform interface across every app" is the only defensible value-add — build that or don't build it.

05

The Frontier Question

Will agent orchestration route real user intent to long-tail indie apps through open discovery — or only to pre-registered, dominant-platform tool registries and enterprise contracts?

This is the single question that flips MCP from "future-proofing" to "over-engineering," and no lens could answer it with evidence. If agents discover and call apps organically, being agent-ready by default is a compounding distribution edge. If — as the 2026 evidence leans — agents route through curated registries and enterprise deals, then agent-accessibility for a niche template app is defensive hygiene at best and unwanted, liability-bearing surface at worst. Everything downstream (how hard to push operability, whether writes are ever worth it) hinges on the answer. Watch which way the MCP registries and client tool-pickers evolve: open crawl-and-discover, or curated allow-lists.
