# 48-Hour MCP Spike — Findings

**Why:** the roast RESHAPE + storm-research named one riskiest assumption to test *before* building an
`agentify`/`build-mcp` skill — namely **(a)** is wrapping the template's `DataStore` as an MCP server mostly
mechanical codegen, and **(b)** does operating an app *through* an agent actually beat opening it? This is
the cheapest test of both, plus a real check of the read-only safety posture.

**What was built** (relocated to `spike-mcp/` at the repo root — a reference blueprint, not a shipped skill; delete if unwanted):
a read-only MCP server over a template-shaped `DataStore` for a *Kept*-like get-paid app (clients, invoices,
payments, tax-reserve settings), driven by a **real MCP client over stdio** (the same protocol Claude Desktop
speaks). Stack: `@modelcontextprotocol/sdk` **1.29.0**, Node 24, zod 4 — no keys, no network, mock data.

---

## Result: the thesis holds. Building this as a skill is low-risk codegen.

### (a) Codegen ratio — measured, not guessed

The design deliberately splits a **generic adapter** (write once, ship with the skill) from a tiny **per-app
descriptor** (what the skill generates for each app). Measured non-comment code lines:

| File | Lines | Nature |
|---|---:|---|
| `adapter.mjs` — `DataStore → MCP` compiler pass | **54** | **Generic. Written once, reused for every app.** Zero app knowledge. |
| `catalog{}` — the per-app entity descriptor | **28** | Declarative (name/methods/key/summary) for 3 entities + 1 singleton. A skill emits this from the `DataStore` TS interface + charter. |
| `server.mjs` — per-app wiring | **9** | Pure boilerplate (import store + catalog, connect stdio). |
| the `DataStore` itself | **0 new** | Already generated by `build-backend`. |

So per app the skill emits **~37 lines of mostly-declarative code**, almost all mechanical. **The only part
that isn't pure codegen is the tool-description prose** (`summary`/`hint`/`keyDesc` — "MCP issue 4: UX" from
the video): ~1–3 lines of domain English per entity, derivable from the charter/field names or a one-line
human review. **Verdict: ~90%+ codegen; the irreducible human part is agent-facing tool descriptions.** The
"compiler pass over a uniform interface" thesis is confirmed — and it's the *only* non-commodity angle
(generic OpenAPI→MCP is already solved by FastMCP/Speakeasy/Supabase-MCP; the edge is the RLS-aware,
DataStore-native, read-first wiring).

### (b) Does agent-operation beat opening the app? — Yes, for *synthesis*; no, for single lookups

Real task asked of the server: *"Where do I stand on getting paid, and am I set for taxes?"* Answered from
**4 read-only tool calls**, correctly:

```
Outstanding (sent + overdue): $5,600 across 4 invoices
    of which OVERDUE:         $2,350 — chase these first
Client who owes the most:     Cedar Realty Group ($2,600)
Collected so far:             $5,800
Tax reserve @ 25%:            should be $1,450, set aside $900
⚠ You are $550 SHORT on your tax set-aside.
```

- **The win is real for cross-entity synthesis.** That single question spans invoices (filtered by status),
  clients (to name the top debtor), payments, and settings — in the app it's 3–4 screens plus mental math;
  through an agent it's one sentence. Aggregate/"where do I stand" questions in natural language are where
  agent-operation genuinely beats navigating UI.
- **The tool surface was self-selecting.** `list_invoices`'s description literally says "use to compute
  outstanding balance… find overdue… total collected income," so tool choice is unambiguous — evidence the
  agent-UX descriptions carry their weight.
- **Honest caveat:** the final summing was done in the probe's JS, standing in for the LLM's own reasoning.
  The spike proves the *data a real task needs is fully reachable through the generated read-only surface*
  and that it takes few calls; the "an LLM picks the tools and reasons" end-to-end is the 10-minute Claude
  Desktop step below. And for single-record lookups ("show invoice 1006") the app is just as fast — the
  agent edge is specifically synthesis + already-in-a-chat context.

### Demand test — a live agent, not the probe (2026-07-03)

Re-ran (b) with a **real Claude agent** (`claude -p --mcp-config --strict-mcp-config`, restricted to the
`kept` MCP tools, no built-ins) so an LLM — not hardcoded JS — picked the tools and reasoned. Two unscripted
questions:

1. *"Where do I stand on getting paid… and taxes?"* → the agent autonomously called `list_invoices`,
   `list_clients`, `get_settings` (3 calls) and returned an answer **richer than the probe or the app**: it
   split outstanding into sent ($3,250) vs overdue ($2,350), named the exact invoice to chase first
   (2026-006), and flagged the $550 tax shortfall with a "top it up" nudge.
2. *"Find my most overdue invoice, get the client's contact, and draft a payment-reminder email."* → the
   agent called `list_invoices` + `get_client` (drill-down, 2 calls), computed "~33 days overdue," and
   **produced a ready-to-send reminder email** — an artifact the app cannot generate at all.

**Signal:** for synthesis and "do-something-with-the-data" tasks, agent-operation clearly beats opening the
app — tool selection was autonomous and correct, and read-only data + an LLM yields *new* capabilities
(prioritize, draft) with zero write access. **What this does NOT prove:** that real end-users will *point*
their agent at a niche app in the wild — that's distribution/discovery (the frontier question), a market
question, not a usefulness one. Usefulness-when-connected is now answered: **yes.**

### Safety posture — read-only *by construction*, verified

- `list_tools` returns **7 tools, every one annotated `readOnlyHint: true`**; **0 write/mutating tools**.
- A write attempt is impossible, not just discouraged: `call create_invoice` → `isError: -32602 "Tool not
  found"`, and the invoice count was **8 before and 8 after**. The generic adapter has *no code path* that
  registers a create/update/delete tool, so the capability physically isn't present.
- This is the concrete resolution of the roast's "attended-for-irreversible" collision: **a read-only server
  cannot take an irreversible action.** Writes would be a separate, opt-in adapter gated behind confirmation
  — deliberately absent here.
- The heavier security surface storm-research flagged (OAuth, remote endpoints, the "lethal trifecta")
  belongs to the *remote/Tier-1* deployment, which stays behind the keyed go-live checklist. The Tier-0
  deliverable (local stdio + mock data) is genuinely keyless and safe.
- Skill-authoring note: MCP tool errors come back as `isError` **results**, not thrown exceptions — the
  skill's generated tests must assert on `isError`, not rely on try/catch.

### Spec / transport note

The tool layer is transport-agnostic: the same `adapter.mjs` + `catalog` retarget a **remote Streamable
HTTP + OAuth** deployment (the 2026-07-28 final spec) by swapping only `StdioServerTransport`. So "build to
the new spec" is an isolated transport concern, not a rewrite of the codegen — de-risking the timing worry.

---

## What this means for the `build-mcp` skill design

1. **Ship the 54-line adapter as a fixed asset in the skill.** Per-app, generate only the ~28-line catalog
   from the `DataStore` interface + charter, plus 9 lines of wiring. This is a genuine compiler pass.
2. **Read-only tools are the whole Tier-0 deliverable.** Default to `list_/get_`. Writes are a later,
   opt-in, confirmation-gated adapter routed through domain/service functions (never raw `DataStore.create`).
3. **The one human-in-the-loop input is tool-description prose.** Derive a first draft from the charter;
   flag it for a quick review. That's the "agent-UX" craft — everything else is mechanical.
4. **Graceful-off is honest here:** the skill scaffolds a keyless local stdio server over the MockStore that
   runs immediately; the remote go-live (Streamable HTTP + OAuth + SupabaseStore) is the one keyed step the
   user owns — same pattern as `build-backend`/`deploy`.

**Net:** the spike removes the technical risk from piece (1) of the RESHAPE. The remaining open question is
still **demand** (the briefing's frontier question — will agents route real users to a long-tail app?), which
code can't answer. Cheapest way to feel that out: the 10-minute Claude Desktop hookup below, tried on a real
day's data.

## Run it yourself (the true end-to-end)

The spike now lives at `spike-mcp/` in the repo root (uncommitted throwaway — see its `README.md`). From
that folder:

```bash
npm install           # already done in this session
npm run probe         # launches the server + drives it (the output above)
```

To let a real LLM agent pick the tools and reason (the genuine test of (b)), add it to
**Claude Desktop** (`claude_desktop_config.json` → Settings ▸ Developer ▸ Edit Config) or to
**Claude Code** (a project `.mcp.json`), then restart the client:

```json
{
  "mcpServers": {
    "kept": {
      "command": "node",
      "args": ["C:/GIT/hma_project_foundation/spike-mcp/src/mcp/server.mjs"]
    }
  }
}
```

Then ask, in plain English: *"Using the kept tools, where do I stand on getting paid, who owes me the most,
and am I setting aside enough for taxes?"* — and judge whether that beats opening the app.
