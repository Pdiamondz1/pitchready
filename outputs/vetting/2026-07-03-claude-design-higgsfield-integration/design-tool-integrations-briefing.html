01

The 60-Second Summary

Of the three tools, only Google Stitch — the incumbent the template already uses — exposes an inbound, headless-drivable surface that emits a portable design-system file (its open-sourced DESIGN.md), and even that ships from an explicitly unsupported Google Labs project. Claude Design (launched April 17, 2026, Pro-plan and up) is confirmed interactive-only: its "MCP connectors" are outbound export destinations (Canva/PDF/PPTX/HTML), its Claude Code handoff is a manual "you can pass" action, and it consumes a design system by reading your codebase rather than emitting one — so a skill cannot drive it at all today. Higgsfield does have a real first-party API + MCP (correcting a widely-repeated claim), but it outputs only raster image/video — no tokens, type, or palette — making it an asset generator, not a design-system tool; it was also the source video's paid affiliate (a creator program paying up to $2,500/video). The durable integration in this space is a format, not a vendor: every named AI-design tool churns — Galileo AI was acqui-hired into Google Stitch within ~12 months, Create React App was sunset, and the W3C Design Tokens spec went stable in Oct 2025 precisely to kill vendor lock-in. The template's own wiki/design-system.md is already that portable target; the honest move is to make it a vendor-neutral contract and treat every tool — Stitch included — as an optional, keyed writer into it.

03

The Hidden Connection

The pitch says "add two powerful tools." The evidence says one can't be driven and the other makes the wrong artifact. Those look like two separate disappointments.

They're the same finding seen twice: generation was never the bottleneck — portability was. The tool that "wins" here (Stitch) wins only because it also emits a portable file (DESIGN.md); Claude Design fails not because it's weak but because it keeps its output inside a walled canvas; Higgsfield fails because its output isn't a design system at all. Every case is decided by the same axis: does it produce a portable, distillable artifact the template already knows how to read?

The template's wiki/design-system.md is the actual asset. Every design tool is only as valuable as its ability to write into that format — so build the socket, not the plug.

The assumption this briefing rests on (and the missing 6th lens)

All five lenses argued about which tool to wire in and how. None asked the user-value question: for a solo dev scaffolding a Tier-0 app, is AI generation of a look-and-feel even the felt gap — or does the existing interview already produce a good-enough design-system.md, making every one of these tools a nice-to-have?

If the interview is the real product (the Buyer strongly implied it is — "I want tokens and a Tailwind config, not a moodboard"), then the priority inverts: the durable work is hardening the format and the offline interview, and "tool integrations" are a distant, optional layer. That single omission could flip this from "which tools to add" to "add none; sharpen the contract."

04

What To Actually Do Differently

For the template maintainer wiring the design step — specific moves, not principles:

01
Generalize the Stitch wiring into a vendor-neutral "design-tool" role.

One paragraph in define-design + a doc table: the contract is "any tool that can populate wiki/design-system.md via Generate→Export→Distill." Vendors become swappable examples, not hardcoded steps — the pattern that survived every churn cycle (Finding 2).

02
Add Claude Design as a documented MANUAL option, honestly framed.

Interactive claude.ai (Pro+), outbound-only, no automation: "use it to explore a look, then describe or paste the result back into the interview." Zero new prompts on the keyless default path (respects the Buyer's onboarding-bloat objection). Never claim it can be driven — it can't (Finding 1).

03
Do NOT wire Higgsfield into define-design.

It's raster-only (Finding 3) — wrong artifact for a design system. If assets ever matter, capture a generic, off-by-default "asset-generation tool" role in a later imagery tier, and reconcile the "no stock photography" voice first. Naming it now imports a paid, affiliate-driven dependency for ~zero indie uptake.

04
Make the format speak a standard: align design-system.md with W3C Design Tokens and/or Stitch's DESIGN.md.

The W3C tokens spec went stable (2025.10) to be the neutral interchange (Finding 2). Speaking a standard — not a vendor dialect — is what makes the socket durable and lets any tool plug in.

05
Keep the existing Stitch MCP, but re-document it honestly and pin it.

It's a Google Labs project with no stability SLA and a third-party package alongside it (Finding 4). Note it can break, pin the package, and keep it strictly optional/keyed — exactly the graceful-off posture the template already uses.

06
Add a one-line honesty note to the skill and docs.

"These external tools change monthly; we integrate the format, not the vendor." It's the methodology-as-moat framing, and it inoculates future maintainers against re-litigating the next "just dropped 2.0" video.

05

The Frontier Question

Will Anthropic ship an inbound programmatic/MCP surface for Claude Design — turning its "over the coming weeks" promise into a real API a skill can drive?

It matters because Claude Design is the one tool ecosystem-aligned with this template (same Anthropic account, same MCP world). If an inbound surface + a portable export land, Claude Design becomes a genuinely native, automatable design-system provider — and the vendor-neutral adapter recommended here is exactly what lets you add it in an afternoon instead of re-architecting. No lens could answer this (the capability doesn't exist yet), which is why building the socket now is the hedge that pays off no matter which way Anthropic goes.
